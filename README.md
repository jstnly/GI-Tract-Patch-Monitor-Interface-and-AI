# GI Tract Patch Monitor — Dashboard (Prototype)

A nurse-facing dashboard for **GI-tract patch monitors** used on babies in a DICU
(digestive intensive care unit). Each patch's three sensors track gut motility —
contraction frequency & strength, MMC (migrating motor complex) duration and
timing, and a bioimpedance **coordination number** — compares them to normal
reference ranges, and surfaces:

- an at-a-glance **Motile Probability** per baby — a 0–100% likelihood the gut is
  motile (high = healthy), a score rather than a diagnosis,
- a **red outline + alert** when a monitor's motile probability drops into a
  concerning band, so a nurse knows to check on that baby,
- **per-baby baseline calibration** — each metric shows the baby's own baseline
  and its % change, and the motile probability is judged against that baseline
  (a new patch shows "Calibrating baseline…" first),
- a **Motility Gain** box: current Motility Index, resting baseline, and **Gain =
  current MI ÷ baseline MI** — a healthy gut ramps ~**2–3×** after a stimulus
  (feeding),
- per-metric **sensor labels** and a **multi-sensor confidence** indicator (rises
  when the three sensors agree),
- an **AI signal-quality check** that flags patch-placement or motion artifacts
  (baby crying / laughing / moving) so nurses know when readings may be
  unreliable — data-quality analysis only, never a diagnosis,
- a **feeding recommendation** (feed / hold / review with clinician) driven by
  the motility metrics and Gain,
- a one-click **detail view** with the full spectrum of metrics and trends,
- **add / remove** monitors at any time.

> ⚠️ **Prototype with simulated data.** The patch hardware does not exist yet, so
> all readings are generated by a built-in simulator. The clinical thresholds are
> chosen to be plausible and internally consistent **for demonstration only** —
> they are not validated for clinical use. The data layer is built behind a clean
> interface so a real patch feed can replace the simulator without touching the UI.

## Run it with no install (just open the file)

`npm run build` produces **one self-contained file** at `dist/index.html` — all
HTML, CSS and JavaScript inlined. **Double-click that file** and it opens and runs
in your default browser. No Node, no server, nothing to install. You can copy it
to a USB stick, email it, or drop it on any hospital PC and it just works.

```bash
npm install      # once
npm run build    # produces dist/index.html
# then double-click dist/index.html  (rename it to anything you like)
```

## Run it for development (live reload)

Requires [Node.js](https://nodejs.org) 18+ (developed on Node 25).

```bash
npm install
npm run dev
```

Open the printed `http://localhost:5173` URL. You'll see a grid of monitors that
update live; one baby starts in an alert state and another drifts into alert on
its own after a minute or so.

### Scripts

| Command | What it does |
| --- | --- |
| `npm run dev` | Start the Vite dev server (hot reload). |

## Using the demo

- **Click any monitor box** to open its pop-out detail window: motile probability
  → feeding advice → all metrics → trends → nurse notes. The windows are draggable and
  resizable, and several can be open at once to compare babies (e.g. twins).
- **Add monitor** (top-right) pairs a new monitor; pick a starting state
  (Normal / Watch / Alert / Random) to seed the demo.
- **Remove** a monitor from inside its detail window (with an inline confirm).
- **Demo** toggle (top bar) reveals global controls: add a random monitor, push a
  calm monitor into a slow drift toward Alert, or pause/resume the simulation.
- Inside a monitor's detail drawer, the collapsible **Demo controls** can force
  that monitor to Normal / Watch / Alert, or back to **Auto** (live drift).

Motile-probability bands: **Normal** 61–100%, **Watch** 31–60% (amber ring),
**Alert** 0–30% (**red ring + entry in the alert banner**). Status is always shown
with an icon and a word as well as colour, so it never relies on colour alone.

## How it's built

- **Vite + React + TypeScript**, plain CSS Modules with a design-token file
  (`src/styles/tokens.css`) — white-dominant with a blue accent. No UI-component
  or charting dependencies (charts are hand-rolled SVG).
- The UI is a **pure function of monitor data**. All clinical logic and the
  simulation live behind a typed boundary.

```
src/
  types/monitor.ts            UI data contract (Monitor, Metric, Abnormality, …)
  lib/
    datasource/               the swappable boundary
      DataSource.ts           interface the UI depends on
      SimulatedDataSource.ts  drives the simulation on a timer
    engine/                   pure, unit-tested clinical logic
      config.ts               every tunable constant (thresholds, bands, profiles)
      abnormalities.ts        metric values → flagged abnormalities (feeds feeding)
      risk.ts                 deviation from baseline → risk % (inverted to motile prob.) + band
      motility.ts             Motility Index + Gain (current MI ÷ baseline MI)
      feeding.ts              metrics + gain → feeding recommendation
      simulation.ts           live, drifting, plausible data generation
  state/                      React context + reducer + tick loop
  components/                 layout · grid · alerts · detail · add · ui
```

### Connecting real patches later

Implement the `DataSource` interface (`src/lib/datasource/DataSource.ts`) for the
real feed (e.g. a `RealtimeDataSource` over WebSocket / BLE) and return it from
`createDataSource()` in `src/lib/datasource/index.ts`. The UI requires no changes;
demo-only controls disappear automatically when the source omits them.

To tune the clinical model (normal ranges, abnormality thresholds, risk band
cut-offs, feeding rules, drift behaviour), edit `src/lib/engine/config.ts` — it is
the single source of truth and is covered by `src/lib/engine/engine.test.ts`.

## Shipping

This builds to plain static files, so it's free to run and free to host.

- **Single file (no install):** `npm run build` → `dist/index.html` is one
  self-contained file. Double-click to open it locally, or drop it on any static
  host (Netlify, Vercel, GitHub Pages, hospital intranet). (Single-file output is
  produced by `vite-plugin-singlefile`; remove that plugin in `vite.config.ts` for
  a conventional split/cached multi-file build when you move to production hosting.)
- **Downloadable desktop app (future):** wrap this build with
  [Tauri](https://tauri.app) to produce a small native `.exe` / `.msi` installer.
  Tauri is free and open-source; it reuses the OS web view, so the binary stays a
  few MB rather than the ~100 MB of an Electron app. This was intentionally left
  out of the first prototype to avoid adding a Rust toolchain.

## Not in this prototype

Authentication, a persistence/backend layer, multi-unit views, audit logging,
push/sound notifications, and real clinical validation of thresholds. The
`DataSource` boundary is already in place for the real patch feed.
